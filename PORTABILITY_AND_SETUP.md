# Portability and Setup Guide

This guide explains how to move this project to another machine and recreate the same setup.

---

## 1) Which files to transfer

Transfer the full project folder **except** generated/local files:

Do transfer:
- `extension.js`
- `package.json`
- `package-lock.json`
- `src/`
- `.vscode/` (optional but useful for debug config)
- markdown docs (`README.md`, `CHANGES_DOCUMENTATION.md`, etc.)
- `.env.example`

Do not transfer:
- `node_modules/`
- `.vscode-test/`
- local log/cache files
- secret-filled `.env` (unless securely moved)

---

## 2) Recreate setup on a new machine

1. Install **Node.js** (LTS recommended).
2. Install **VS Code**.
3. Copy project folder to new machine.
4. Open folder in VS Code.
5. Run:
   - `npm install`
6. Create local env file:
   - copy `.env.example` -> `.env`
   - set `TAVILY_API_KEY=...`
7. Press `F5` to open Extension Development Host.
8. In dev host chat, use `@dsxchangeassist`.

---

## 3) `.gitignore` explanation

Current `.gitignore` entries and why:

- `node_modules/`
  - dependency folder generated by `npm install`.
- `.vscode-test/`
  - generated by VS Code extension test tooling.
- `*.vsix`
  - packaged extension artifacts.
- `out/`, `dist/`
  - build output folders.
- `.env`, `.env.*`
  - local environment and secrets.
- `!.env.example`
  - keep sample env template committed.
- `npm-debug.log*`, `yarn-debug.log*`, `yarn-error.log*`, `pnpm-debug.log*`
  - package manager log noise.
- `.DS_Store`, `Thumbs.db`
  - OS-generated metadata files.

---

## 4) Why this matters

Good `.gitignore` hygiene helps by:
- preventing accidental secret commits,
- avoiding large generated files in git history,
- keeping pull requests focused on real source changes.

---

## 5) Quick verification checklist after moving

- `npm run check` passes.
- `.env` exists with valid `TAVILY_API_KEY`.
- `F5` launches extension host.
- `@dsxchangeassist` appears in Chat view.
- `/websearch <query>` works.
- edit flow works with `/edit` requests and direct file edits.
